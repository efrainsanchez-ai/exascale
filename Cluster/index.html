1. Introduction

Oracle Exadata Exascale Database Service on Oracle Cloud Infrastructure (OCI) enables elastic scaling of Oracle Real Application Clusters (RAC) databases. This document explains the architectural transition from a single-node RAC deployment to a two-node RAC deployment, and the same service-driven pattern can be extended from one node to a ten-node cluster when the tenancy, limits, and workload design support that target shape.

The Exascale operating model makes that scale-out path automation-oriented: add database server capacity through the cloud service workflow, let Oracle-managed infrastructure integrate the node into the VM cluster and RAC stack, then validate services, listeners, SCAN routing, instance placement, workload distribution, and application connection pools before increasing production traffic.

In an on-premises environment, moving from a small RAC footprint to a ten-node cluster usually means a larger project: hardware procurement, rack and power planning, network capacity, storage layout, firmware and operating system alignment, Grid Infrastructure expansion, database home consistency, patch coordination, failover testing, and application certification.

The cost profile is different as well. On-premises expansion often requires capital purchase and reserved data center capacity before the workload fully needs it. Exascale shifts more of that decision into an elastic service model where teams can scale closer to demand, but the financial benefit still depends on disciplined capacity planning, workload consolidation, license posture, and active monitoring after the change.

3. RAC Availability Concepts

In this guide, the RAC database is CDB01, the pluggable database is PDB01, and the application enters through the service CDB01_PDB01.paas.oracle.com. After scaling, CDB011 and CDB012 are the RAC instances that can host that service. Applications should use the service name and SCAN address, not instance-specific connect strings, so routing can change without changing client configuration.

SCAN is the stable cluster entry point. DNS resolves the SCAN name to SCAN VIPs, SCAN listeners receive the request, and the cluster routes the connection to a local listener where the requested service is currently available. This hides node placement from clients and allows services to remain the operational boundary for routing, failover, and workload management.

Fast Application Notification (FAN) publishes planned and unplanned service events, while Oracle Notification Service (ONS) transports those events to aware clients and pools. Fast Connection Failover (FCF) lets integrated pools quickly discard dead connections and create replacements. Runtime Load Balancing (RLB) uses RAC service advisories so new work favors the instance currently meeting the selected service goal.

SCAN, FAN, ONS, FCF, and RLB improve connection routing and recovery, but they do not replay work by themselves. Application Continuity (AC) and Transparent Application Continuity (TAC) add replay behavior for recoverable requests when the client driver, service attributes, and application state make replay safe.

SCANProvides the stable cluster access name for service-based connections.It does not replay SQL or preserve in-flight work.
FAN / ONSPublishes and transports RAC service and node events.It does not fix applications that keep using closed connections.

4. Replay, Request Boundaries, and Memory

Replay means the client stack can rebuild a recoverable database session and repeat a safe logical unit of work after an outage. The key word is safe: the driver must know the start and end of a request, whether the last commit completed, and whether the session state can be restored consistently.

AC and TAC depend on service attributes, RAC services, FAN/ONS events, RLB-aware clients, and replay-capable drivers or pools such as Oracle JDBC with UCP. TAC automates more of the state tracking, while AC is more explicit. Both still require application validation because package globals, temporary tables, altered session state, open cursors, external side effects, and long conversations can make replay unsafe or less useful.

Failover also has a memory cost. Every replacement or replayed session consumes process memory on the target instance. During a CDB011 outage, CDB012 may carry surviving sessions plus replacement sessions, including PGA, UGA, cursor memory, and package state. Capacity planning must include this failover posture, not only normal balanced operation.

  • Mark request boundaries or use pools and drivers that can identify them reliably.
  • Keep OLTP requests short, idempotent where possible, and free of hidden session affinity.
  • Validate replay behavior with the real driver, pool, service attributes, and workload pattern.
  • Measure sessions, processes, and memory on CDB011 and CDB012 during both normal routing and failure simulation.

5. OLTP Runtime Load Balancing and Connection Model

Runtime load balancing is a service advisory mechanism consumed by integrated connection pools. It does not move existing sessions. Instead, it publishes service-quality advice that helps supported pools choose the best RAC instance when they borrow or create a connection for new work.

This section modifies the existing CDB01_PDB01.paas.oracle.com OLTP service to use SHORT connection load balancing and SERVICE_TIME runtime load balancing. The service name stays the same, so application clients can continue using the same connect descriptor or TNS alias while the service begins publishing response-time-oriented RAC advisories.

SERVICE_TIME and THROUGHPUT are both Runtime Load Balancing goals, but they optimize for different signals. SERVICE_TIME favors the instance with the best observed response time for the service. It is usually the clearer OLTP choice when requests are short, user-facing, and latency-sensitive because the advisory prefers the instance completing individual requests quickly.

THROUGHPUT favors the instance completing the highest rate of work. It can be useful when the priority is total work completed per unit of time, but it can be less intuitive for latency-sensitive OLTP because a high-throughput instance is not always the lowest-latency instance for the next short request. For mixed or stateful applications, validate both goals with the real pool, driver, and workload before changing production behavior.

Runtime load balancing works best for stateless request models where connections are borrowed from the pool, used for a short request, and returned to the pool. Enable Fast Connection Failover and use UCP, WebLogic, OCI, or ODP.NET pools that consume FAN and RAC load balancing advisory events.

Bash
Objective: modify the existing OLTP service to publish SHORT connection load balancing and SERVICE_TIME runtime load balancing advisories for integrated pools.
srvctl modify service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com \
-clbgoal SHORT \
-rlbgoal SERVICE_TIME
Bash
Objective: stop, start, and check the service so the updated CLB and RLB attributes are reloaded across the cluster.

7. Application Continuity (AC)

Example configuration

Application Continuity protects user work by rebuilding a recoverable database session and replaying a safe request after planned maintenance, a network interruption, or an instance failure. It is most useful when the application uses request boundaries from a supported pool or driver, because replay must start and stop at a known logical unit of work.

The TRANSACTION failover type enables transaction replay behavior for the service. commit_outcome helps the client determine whether the last commit completed, while replay_init_time and retention define the replay initiation window and how long commit outcome information is retained.

Example: Server Control Utility (SRVCTL) manages Oracle RAC database services. In this guide, CDB01 is the Container Database (CDB), and CDB01_PDB01.paas.oracle.com is the Pluggable Database (PDB) service.

Bash
Objective: apply Application Continuity service attributes to the RAC service used by the application.
srvctl modify service \
-db CDB01 \
-service CDB01_PDB01.paas.oracle.com \
-failovertype TRANSACTION \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

8. Transparent Application Continuity (TAC)

Recommended configuration

Transparent Application Continuity is the automatic mode of Application Continuity. It tracks session and transactional state for supported clients so the database session can be recovered after a recoverable outage without requiring the DBA to know the application code path.

Use TAC for applications where request boundaries can be discovered safely and where the client stack is replay-aware, such as JDBC/UCP or supported Oracle drivers. The AUTO failover type enables TAC, and session_state DYNAMIC lets Oracle track session state changes instead of requiring a fixed static model.

Recommended configuration: SRVCTL applies the TAC service attributes to the CDB01 database and CDB01_PDB01.paas.oracle.com service used in this example.

Bash
Objective: apply Transparent Application Continuity service attributes to the RAC service used by the application.
srvctl modify service \
-db CDB01 \
-service CDB01_PDB01.paas.oracle.com \
-failovertype AUTO \
-session_state DYNAMIC \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

Operational checklist

1. Introducción

Oracle Exadata Exascale Database Service en Oracle Cloud Infrastructure (OCI) permite el escalado elástico de bases de datos Oracle Real Application Clusters (RAC). Este documento explica la transición arquitectónica desde un despliegue RAC de un solo nodo hacia un despliegue RAC de dos nodos, y el mismo patrón basado en servicios puede extenderse de un nodo a un clúster de diez nodos cuando la tenencia, los límites y el diseño de carga soportan esa forma objetivo.

El modelo operativo de Exascale convierte ese camino de escalamiento en un proceso orientado a automatización: agregar capacidad de database server mediante el flujo del servicio cloud, dejar que la infraestructura administrada por Oracle integre el nodo al VM cluster y al stack RAC, y luego validar servicios, listeners, ruteo SCAN, colocación de instancias, distribución de carga y pools de conexión de aplicación antes de aumentar tráfico productivo.

En un ambiente on-premises, pasar de una huella RAC pequeña a un clúster de diez nodos normalmente implica un proyecto mayor: procurement de hardware, planeación de rack y energía, capacidad de red, layout de almacenamiento, alineación de firmware y sistema operativo, expansión de Grid Infrastructure, consistencia de database homes, coordinación de patching, pruebas de failover y certificación de aplicaciones.

El perfil de costo también cambia. La expansión on-premises suele requerir compra de capital y capacidad reservada de centro de datos antes de que la carga la necesite por completo. Exascale mueve más de esa decisión hacia un modelo de servicio elástico donde los equipos pueden escalar más cerca de la demanda, pero el beneficio financiero aún depende de planeación disciplinada de capacidad, consolidación de cargas, postura de licenciamiento y monitoreo activo después del cambio.

3. Conceptos de disponibilidad RAC

En esta guía, la base RAC es CDB01, la pluggable database es PDB01 y la aplicación entra por el servicio CDB01_PDB01.paas.oracle.com. Después del escalado, CDB011 y CDB012 son las instancias RAC que pueden hospedar ese servicio.

SCAN es el punto estable de entrada al clúster. DNS resuelve el nombre SCAN a VIPs de SCAN, los listeners SCAN reciben la solicitud y el clúster enruta la conexión hacia un listener local donde el servicio está disponible. Esto evita que los clientes dependan de una instancia específica.

FAN publica eventos planeados y no planeados de servicios y nodos; ONS transporta esos eventos a clientes y pools compatibles. FCF permite que los pools descarten conexiones muertas y creen reemplazos. RLB usa advisories del servicio RAC para que el nuevo trabajo favorezca la instancia que cumple mejor el objetivo configurado.

SCAN, FAN, ONS, FCF y RLB mejoran ruteo y recuperación, pero no reproducen trabajo por sí solos. AC y TAC agregan replay para solicitudes recuperables cuando el driver, los atributos del servicio y el estado de la aplicación lo permiten.

FAN / ONSPublica y transporta eventos RAC.No corrige aplicaciones que reutilizan conexiones cerradas.
FCFInvalida conexiones muertas en pools integrados.No decide si una transacción puede repetirse.

4. Replay, límites de solicitud y memoria

Replay significa que el stack cliente puede reconstruir una sesión recuperable y repetir una unidad lógica de trabajo segura después de una interrupción. El driver debe conocer el inicio y fin de la solicitud, si el último commit terminó y si el estado de sesión puede restaurarse de forma consistente.

AC y TAC dependen de atributos de servicio, servicios RAC, eventos FAN/ONS, clientes con RLB y drivers o pools compatibles con replay como Oracle JDBC con UCP. TAC automatiza más seguimiento de estado; AC es más explícito. Ambos requieren validación de aplicación.

El failover también consume memoria. Cada sesión reemplazada o reproducida usa memoria de proceso en la instancia destino. Durante una falla de CDB011, CDB012 puede cargar sesiones sobrevivientes y sesiones reemplazo, incluyendo PGA, UGA, cursores y estado de paquetes.

  • Marque límites de solicitud o use pools y drivers que puedan identificarlos.
  • Mantenga solicitudes OLTP cortas y sin afinidad oculta de sesión.
  • Valide replay con el driver, pool, atributos de servicio y workload reales.
  • Mida sesiones, procesos y memoria en CDB011 y CDB012 durante operación normal y simulaciones.

5. Runtime Load Balancing OLTP y modelo de conexión

Runtime load balancing es un mecanismo de service advisory consumido por pools de conexión integrados. No mueve sesiones existentes. Publica señales de calidad de servicio para que los pools soportados elijan la mejor instancia RAC cuando toman o crean una conexión para trabajo nuevo.

Esta sección modifica el servicio OLTP existente CDB01_PDB01.paas.oracle.com para usar connection load balancing SHORT y runtime load balancing SERVICE_TIME. El nombre de servicio no cambia, por lo que los clientes pueden seguir usando el mismo descriptor o alias TNS mientras el servicio publica advisories orientados a tiempo de respuesta.

SERVICE_TIME y THROUGHPUT son objetivos de Runtime Load Balancing, pero optimizan señales diferentes. SERVICE_TIME favorece la instancia con mejor tiempo de respuesta observado para el servicio. Suele ser la opción más clara para OLTP cuando las solicitudes son cortas, interactivas y sensibles a latencia.

THROUGHPUT favorece la instancia que completa la mayor tasa de trabajo. Puede ser útil cuando la prioridad es el volumen total completado por unidad de tiempo, pero puede ser menos intuitivo para OLTP sensible a latencia porque la instancia de mayor throughput no siempre es la de menor latencia para la siguiente solicitud corta. Para aplicaciones mixtas o stateful, valide ambos objetivos con el pool, driver y workload reales.

Runtime load balancing funciona mejor para modelos stateless donde las conexiones se toman del pool, se usan durante una solicitud corta y se devuelven al pool. Habilite Fast Connection Failover y use UCP, WebLogic, OCI u ODP.NET con eventos FAN y RAC load balancing advisory.

Objetivo: modificar el servicio OLTP existente para publicar connection load balancing SHORT y runtime load balancing SERVICE_TIME para pools integrados.

-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com \
-clbgoal SHORT \
-rlbgoal SERVICE_TIME

7. Application Continuity (AC)

Configuración de ejemplo

Application Continuity protege el trabajo del usuario al reconstruir una sesión de base de datos recuperable y reproducir una petición segura después de mantenimiento planificado, una interrupción de red o una falla de instancia. Es más útil cuando la aplicación usa límites de petición desde un pool o driver compatible, porque el replay debe iniciar y terminar en una unidad lógica de trabajo conocida.

El tipo de failover TRANSACTION habilita comportamiento de replay de transacciones para el servicio. commit_outcome ayuda al cliente a determinar si el último commit se completó, mientras que replay_init_time y retention definen la ventana de inicio de replay y cuánto tiempo se conserva la información de resultado de commit.

Ejemplo: Server Control Utility (SRVCTL) administra servicios de base de datos Oracle RAC. En esta guía, CDB01 es la Container Database (CDB), y CDB01_PDB01.paas.oracle.com es el servicio de Pluggable Database (PDB).

Bash
Objetivo: aplicar atributos de Application Continuity al servicio RAC usado por la aplicación.
srvctl modify service \
-db CDB01 \
-service CDB01_PDB01.paas.oracle.com \
-failovertype TRANSACTION \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

8. Transparent Application Continuity (TAC)

Configuración recomendada

Transparent Application Continuity es el modo automático de Application Continuity. Rastrea el estado de sesión y transaccional para clientes compatibles, de modo que la sesión de base de datos pueda recuperarse después de una interrupción recuperable sin requerir que el DBA conozca la ruta de código de la aplicación.

Use TAC para aplicaciones donde los límites de petición puedan descubrirse de forma segura y donde la pila cliente sea compatible con replay, como JDBC/UCP o drivers Oracle soportados. El tipo de failover AUTO habilita TAC, y session_state DYNAMIC permite que Oracle rastree cambios de estado de sesión en lugar de requerir un modelo estático fijo.

1. Introdução

O Oracle Exadata Exascale Database Service no Oracle Cloud Infrastructure (OCI) permite escalonamento elástico de bancos de dados Oracle Real Application Clusters (RAC). Este documento explica a transição arquitetônica de uma implantação RAC de nó único para uma implantação RAC de dois nós, e o mesmo padrão baseado em serviços pode ser estendido de um nó para um cluster de dez nós quando a tenancy, os limites e o desenho da carga suportam essa forma alvo.

O modelo operacional do Exascale torna esse caminho de escala orientado por automação: adicionar capacidade de database server pelo fluxo do serviço cloud, permitir que a infraestrutura gerenciada pela Oracle integre o nó ao VM cluster e ao stack RAC, e então validar serviços, listeners, roteamento SCAN, posicionamento de instâncias, distribuição de carga e pools de conexão da aplicação antes de aumentar o tráfego produtivo.

Em um ambiente on-premises, sair de uma pegada RAC pequena para um cluster de dez nós normalmente vira um projeto maior: procurement de hardware, planejamento de rack e energia, capacidade de rede, layout de armazenamento, alinhamento de firmware e sistema operacional, expansão do Grid Infrastructure, consistência de database homes, coordenação de patching, testes de failover e certificação de aplicações.

O perfil de custo também muda. A expansão on-premises costuma exigir compra de capital e capacidade reservada de data center antes que a carga precise totalmente dela. O Exascale desloca mais dessa decisão para um modelo de serviço elástico em que as equipes podem escalar mais perto da demanda, mas o benefício financeiro ainda depende de planejamento disciplinado de capacidade, consolidação de cargas, postura de licenciamento e monitoramento ativo após a mudança.

3. Conceitos de disponibilidade RAC

Neste guia, o banco RAC é CDB01, o pluggable database é PDB01 e a aplicação entra pelo serviço CDB01_PDB01.paas.oracle.com. Após o escalonamento, CDB011 e CDB012 são as instâncias RAC que podem hospedar esse serviço.

SCAN é o ponto estável de entrada do cluster. DNS resolve o nome SCAN para VIPs de SCAN, os listeners SCAN recebem a solicitação e o cluster roteia a conexão para um listener local onde o serviço está disponível. Isso evita dependência de uma instância específica.

FAN publica eventos planejados e não planejados de serviços e nós; ONS transporta esses eventos para clientes e pools compatíveis. FCF permite que pools descartem conexões mortas e criem substitutas. RLB usa advisories do serviço RAC para que novo trabalho prefira a instância que atende melhor ao objetivo configurado.

SCAN, FAN, ONS, FCF e RLB melhoram roteamento e recuperação, mas não reproduzem trabalho sozinhos. AC e TAC adicionam replay para solicitações recuperáveis quando driver, atributos do serviço e estado da aplicação tornam o replay seguro.

ConceitoPropósitoO que não faz sozinho
SCANFornece nome estável de acesso ao cluster.Não reproduz SQL nem preserva trabalho em andamento.
FAN / ONSPublica e transporta eventos RAC.Não corrige aplicações que reutilizam conexões fechadas.
FCFInvalida conexões mortas em pools integrados.Não decide se uma transação pode ser repetida.
RLBGuia novas solicitações para a melhor instância do objetivo do serviço.Não move sessões existentes entre instâncias.
AC / TACPode reproduzir solicitações recuperáveis quando é seguro.Não reproduz trabalho stateful inseguro sem limites corretos.

4. Replay, limites de solicitação e memória

Replay significa que o stack cliente pode reconstruir uma sessão recuperável e repetir uma unidade lógica de trabalho segura após uma interrupção. O driver precisa conhecer início e fim da solicitação, se o último commit concluiu e se o estado da sessão pode ser restaurado de forma consistente.

AC e TAC dependem de atributos de serviço, serviços RAC, eventos FAN/ONS, clientes com RLB e drivers ou pools compatíveis com replay como Oracle JDBC com UCP. TAC automatiza mais rastreamento de estado; AC é mais explícito. Ambos exigem validação da aplicação.

Failover também consome memória. Cada sessão substituída ou reproduzida usa memória de processo na instância destino. Durante uma falha de CDB011, CDB012 pode carregar sessões sobreviventes e substitutas, incluindo PGA, UGA, cursores e estado de pacotes.

  • Marque limites de solicitação ou use pools e drivers que possam identificá-los.
  • Mantenha solicitações OLTP curtas e sem afinidade oculta de sessão.
  • Valide replay com driver, pool, atributos de serviço e workload reais.
  • Meça sessões, processos e memória em CDB011 e CDB012 durante operação normal e simulações.

5. Runtime Load Balancing OLTP e modelo de conexão

Runtime load balancing é um mecanismo de service advisory consumido por pools de conexão integrados. Ele não move sessões existentes. Ele publica sinais de qualidade de serviço para que pools suportados escolham a melhor instância RAC ao obter ou criar uma conexão para novo trabalho.

Esta seção modifica o serviço OLTP existente CDB01_PDB01.paas.oracle.com para usar connection load balancing SHORT e runtime load balancing SERVICE_TIME. O nome do serviço permanece igual, então clientes podem continuar usando o mesmo descritor ou alias TNS enquanto o serviço publica advisories orientados a tempo de resposta.

SERVICE_TIME e THROUGHPUT são objetivos de Runtime Load Balancing, mas otimizam sinais diferentes. SERVICE_TIME favorece a instância com melhor tempo de resposta observado para o serviço. Normalmente é a opção mais clara para OLTP quando as solicitações são curtas, interativas e sensíveis a latência.

THROUGHPUT favorece a instância que conclui a maior taxa de trabalho. Pode ser útil quando a prioridade é o volume total concluído por unidade de tempo, mas pode ser menos intuitivo para OLTP sensível a latência porque a instância de maior throughput nem sempre é a de menor latência para a próxima solicitação curta. Para aplicações mistas ou stateful, valide ambos os objetivos com pool, driver e workload reais.

Runtime load balancing funciona melhor para modelos stateless onde conexões são obtidas do pool, usadas por uma solicitação curta e devolvidas ao pool. Habilite Fast Connection Failover e use UCP, WebLogic, OCI ou ODP.NET com eventos FAN e RAC load balancing advisory.

Bash

-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com \
-clbgoal SHORT \
-rlbgoal SERVICE_TIME
