Introduction

Oracle Exadata Exascale Database Service on Oracle Cloud Infrastructure (OCI) enables elastic scaling of Oracle Real Application Clusters (RAC) databases. This document explains the architectural transition from a single-node RAC deployment to a two-node RAC deployment running Oracle Database 26ai.

Architecture Evolution

Application Continuity (AC)

Example configuration

Example: Server Control Utility (SRVCTL) manages Oracle RAC database services. The placeholder MYCDB represents the Container Database (CDB), and pdb1 represents a Pluggable Database (PDB) service.

Bash
srvctl modify service \
-db MYCDB \
-service pdb1 \
-failovertype TRANSACTION \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

Transparent Application Continuity (TAC)

Recommended configuration

Recommended configuration: SRVCTL applies the TAC service attributes to the CDB and PDB service placeholders used in this example.

Bash
srvctl modify service \
-db MYCDB \
-service pdb1 \
-failovertype AUTO \
-session_state DYNAMIC \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

Session and Memory Distribution Queries

These Structured Query Language (SQL) queries use Oracle dynamic performance views to compare user sessions and process memory across RAC instances and PDBs.

Sessions per Instance
SQL
SELECT inst_id, COUNT(*)
FROM gv$session
GROUP BY inst_id;
Sessions per PDB and Instance
SQL
SELECT s.inst_id,
       p.name pdb_name,
       COUNT(*) sessions
FROM gv$session s
JOIN v$containers p
  ON s.con_id=p.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,p.name
ORDER BY p.name,s.inst_id;
Memory Usage per PDB and RAC Instance
SQL
SELECT
    s.inst_id,
    c.name pdb_name,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) pga_mb,
    COUNT(*) sessions
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;
Cluster-Wide Memory Distribution
SQL
SELECT
    c.name pdb_name,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) total_pga_mb,
    COUNT(*) sessions
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY c.name
ORDER BY total_pga_mb DESC;
RAC Workload Dashboard
SQL
SELECT
    s.inst_id,
    c.name pdb_name,
    COUNT(*) sessions,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) total_pga_mb,
    ROUND(SUM(p.pga_alloc_mem)/COUNT(*)/1024/1024,2) avg_pga_mb_per_session
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;

Best Practices

Operational checklist
  • Use SCAN exclusively.
  • Use services instead of instance-specific connections.
  • Enable FAN and Runtime Load Balancing.
  • Use Universal Connection Pool (UCP) with TAC.
  • Validate service distribution after scaling.
  • Test rolling maintenance procedures regularly.

Conclusion

A two-node Oracle RAC deployment on Exadata Exascale provides active-active scalability, workload balancing, and high availability. Combining SCAN, FAN, RLB, AC, and TAC delivers a resilient architecture for Oracle Database 26ai workloads.

Introducción

Oracle Exadata Exascale Database Service en Oracle Cloud Infrastructure (OCI) permite el escalado elástico de bases de datos Oracle Real Application Clusters (RAC). Este documento explica la transición arquitectónica desde un despliegue RAC de un solo nodo hacia un despliegue RAC de dos nodos ejecutando Oracle Database 26ai.

Evolución de la arquitectura

Entrada Conexiones de clientes

Los clientes se conectan mediante conexiones basadas en servicios en lugar de endpoints específicos por instancia.

  • Clientes
  • Nombre de servicio
SCAN Capa de acceso Single Client Access Name (SCAN)

La ruta de conexión usa SCAN, resolución Domain Name System (DNS), Virtual IPs (VIPs) de SCAN y listeners SCAN antes de la búsqueda de servicio.

  • Nombre SCAN
  • SCAN DNS
  • VIPs de SCAN
  • Listeners SCAN
Servicio Resolución de servicio

Los servicios resuelven la conexión hacia listeners locales en los nodos RAC disponibles.

  • Servicios
  • Búsqueda de servicio
  • Listeners locales
Base de un nodo Nodo1 / DB26AI1

El Nodo1 hospeda DB26AI1 en el despliegue RAC de un solo nodo y permanece activo después del escalado.

  • Virtual IP (VIP) de nodo
  • Listener local
  • Instancia de base de datos
RAC de dos nodos Nodo2 / DB26AI2

El Nodo2 agrega DB26AI2 para el despliegue RAC de dos nodos.

  • Virtual IP (VIP) de nodo
  • Listener local
  • Instancia de base de datos
Fast Application Notification (FAN)

Incluido en la pila de alta disponibilidad con Oracle Notification Service.

Oracle Notification Service (ONS)

Se muestra con FAN como parte de la lista de componentes SCAN.

Runtime Load Balancing (RLB)

Runtime Load Balancing es parte de los beneficios de RAC de dos nodos y de la configuración de servicios.

Application Continuity (AC) / Transparent Application Continuity (TAC)

Application Continuity y Transparent Application Continuity ofrecen opciones de disponibilidad con replay.

Beneficios

  • Procesamiento activo-activo
  • Mantenimiento rolling

Introdução

O Oracle Exadata Exascale Database Service no Oracle Cloud Infrastructure (OCI) permite escalonamento elástico de bancos de dados Oracle Real Application Clusters (RAC). Este documento explica a transição arquitetônica de uma implantação RAC de nó único para uma implantação RAC de dois nós executando Oracle Database 26ai.

Evolução da arquitetura

Entrada Conexões de clientes

Os clientes se conectam por conexões baseadas em serviços, em vez de endpoints específicos por instância.

  • Clientes
  • Nome do serviço

O caminho de conexão usa SCAN, resolução Domain Name System (DNS), Virtual IPs (VIPs) do SCAN e listeners SCAN antes da busca de serviço.

  • Listeners SCAN

  • Listeners locais
Base de nó único Nó1 / DB26AI1

O Nó1 hospeda DB26AI1 na implantação RAC de nó único e permanece ativo após o escalonamento.

  • Listener local
  • Instância do banco de dados
RAC de dois nós Nó2 / DB26AI2

O Nó2 adiciona DB26AI2 para a implantação RAC de dois nós.

Application Continuity (AC) / Transparent Application Continuity (TAC)

Benefícios

Application Continuity (AC)

Configuração de exemplo

Exemplo: Server Control Utility (SRVCTL) administra serviços de banco de dados Oracle RAC. O marcador MYCDB representa a Container Database (CDB), e pdb1 representa um serviço de Pluggable Database (PDB).

Bash
srvctl modify service \
-db MYCDB \
-service pdb1 \
-failovertype TRANSACTION \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

Transparent Application Continuity (TAC)

Configuração recomendada

Configuração recomendada: SRVCTL aplica os atributos de serviço TAC aos marcadores de CDB e serviço PDB usados neste exemplo.

Bash
srvctl modify service \
-db MYCDB \
-service pdb1 \
-failovertype AUTO \
-session_state DYNAMIC \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

Descritor de conexão Transparent Network Substrate (TNS)

Este descritor do Oracle Net usa o nome de serviço PDB1, o nome de host SCAN e as opções de balanceamento de carga e failover para acesso baseado em serviços.

Consultas de distribuição de sessões e memória

Estas consultas Structured Query Language (SQL) usam visões dinâmicas de desempenho da Oracle para comparar sessões de usuário e memória de processos entre instâncias RAC e PDBs.

Sessões por instância
SQL
SELECT inst_id, COUNT(*)
FROM gv$session
GROUP BY inst_id;
Sessões por PDB e instância
SQL
SELECT s.inst_id,
       p.name pdb_name,
       COUNT(*) sessions
FROM gv$session s
JOIN v$containers p
  ON s.con_id=p.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,p.name
ORDER BY p.name,s.inst_id;
Uso de memória por PDB e instância RAC
SQL
SELECT
    s.inst_id,
    c.name pdb_name,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) pga_mb,
    COUNT(*) sessions
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;
Distribuição de memória em todo o cluster
SQL
SELECT
    c.name pdb_name,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) total_pga_mb,
    COUNT(*) sessions
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY c.name
ORDER BY total_pga_mb DESC;
Dashboard de carga de trabalho RAC
SQL
SELECT
    s.inst_id,
    c.name pdb_name,
    COUNT(*) sessions,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) total_pga_mb,
    ROUND(SUM(p.pga_alloc_mem)/COUNT(*)/1024/1024,2) avg_pga_mb_per_session
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;

Melhores práticas

Lista operacional
  • Use SCAN exclusivamente.
  • Use serviços em vez de conexões específicas por instância.
  • Habilite FAN e Runtime Load Balancing.
  • Use Universal Connection Pool (UCP) com TAC.
  • Valide a distribuição de serviços após o escalonamento.
  • Teste regularmente os procedimentos de manutenção rolling.

Conclusão

Uma implantação Oracle RAC de dois nós no Exadata Exascale fornece escalabilidade ativo-ativo, balanceamento de carga de trabalho e alta disponibilidade. A combinação de SCAN, FAN, RLB, AC e TAC entrega uma arquitetura resiliente para cargas de trabalho do Oracle Database 26ai.

Glossário

Estas definições resumem os acrônimos usados no artigo e apontam para documentação oficial da Oracle para referência mais detalhada.

AC Application Continuity

Alta disponibilidade com capacidade de replay para interrupções recuperáveis quando o replay está habilitado e é seguro.

CDB Container Database

Contêiner multitenant de banco de dados usado nos exemplos de serviço; os serviços PDB são executados dentro dele.

CRSCTL Controle de Clusterware

Utilitário de linha de comando usado para verificar e administrar recursos do Oracle Clusterware.

DNS Domain Name System

Camada de resolução de nomes usada por clientes para resolver nomes como SCAN.

ExaDB-XS Exadata Database Service on Exascale Infrastructure

Opção de serviço cloud da Oracle para Exadata Exascale Database Service no OCI.

FAN Fast Application Notification

Estrutura de notificação de eventos de alta disponibilidade do Oracle RAC para eventos planejados e não planejados.

FCF Fast Connection Failover

Comportamento de failover do cliente que ajuda a reconectar aplicações após mudanças de serviço ou instância.

OCI Oracle Cloud Infrastructure

Plataforma cloud da Oracle para executar recursos do Exadata Exascale Database Service.

ONS Oracle Notification Service

Transporte de notificações usado com eventos FAN para clientes conscientes de Oracle RAC.

PDB Pluggable Database

Contêiner de banco de dados dentro de uma CDB; este artigo usa pdb1 como marcador de serviço.

RAC Real Application Clusters

Arquitetura Oracle em que várias instâncias em servidores de cluster acessam um banco de dados compartilhado.

RLB Runtime Load Balancing

Roteamento consciente de serviços que usa informações de carga de trabalho para direcionar conexões entre instâncias RAC.

SCAN Single Client Access Name

Nome estável de acesso ao cluster usado por clientes em vez de endereços específicos por nó.

SQL Structured Query Language

Linguagem de consulta usada aqui para inspecionar sessões RAC e distribuição de memória de processos.

SRVCTL Server Control Utility

Utilitário de linha de comando da Oracle para administrar bancos de dados RAC, serviços, listeners e recursos relacionados.

TAC Transparent Application Continuity

Modo de Application Continuity que rastreia automaticamente estado de sessão e limites de requisição.

TNS Transparent Network Substrate

Formato de descritor de conexão do Oracle Net usado para definir serviço, host, balanceamento de carga e failover.

UCP Universal Connection Pool

Pool de conexões JDBC da Oracle usado comumente com recursos de alta disponibilidade conscientes de RAC.

VIP Virtual IP

Endereço de rede administrado pelo cluster usado por listeners SCAN e caminhos de acesso de nó.
