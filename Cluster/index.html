Introduction

Oracle Exadata Exascale Database Service on Oracle Cloud Infrastructure (OCI) enables elastic scaling of Oracle Real Application Clusters (RAC) databases. This document explains the architectural transition from a single-node RAC deployment to a two-node RAC deployment running Oracle Database 26ai.

Replay concept, components, and memory implications

Replay conceptReplay means the client stack can rebuild a recoverable database session and repeat a safe logical unit of work after an outage. SCAN and Fast Connection Failover (FCF) reconnect; Application Continuity (AC) and Transparent Application Continuity (TAC) add replay.
Replay componentsService attributes configured with SRVCTL, Oracle RAC services, SCAN listeners, Fast Application Notification (FAN), Oracle Notification Service (ONS), Runtime Load Balancing (RLB), and replay-aware client drivers or pools such as JDBC/UCP.
Session requirementsApplications must connect through services, not fixed instances. After node failure or service relocation, replacement sessions are established on surviving RAC instances, so pools need enough capacity and retry behavior.
Memory implicationsEvery rebuilt or replayed session consumes process memory on the target instance. During failover, surviving nodes can carry extra sessions plus Program Global Area (PGA), User Global Area (UGA), cursor, and package-state memory.
Operational validationCompare gv$session, gv$process, v$containers, and service placement after scaling or failover. Validate balanced placement, acceptable sessions per instance, and enough memory headroom.

11. Recommended OLTP Configuration (Memory Efficient)

For the memory-efficient OLTP recommendation, keep the default connection service and apply session-level failover settings. This avoids creating another service and keeps the application connection target stable while adding basic reconnect behavior.

Bash

-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com \
-failovertype SESSION \
-failovermethod BASIC \
-failoverretry 30 \
-failoverdelay 5

Session and Memory Distribution Queries

These Structured Query Language (SQL) queries use Oracle dynamic performance views to compare user sessions and process memory across RAC instances and PDBs.

Sessions per Instance
SQL
SELECT inst_id, COUNT(*)
FROM gv$session
GROUP BY inst_id;
Sessions per PDB and Instance
SQL
SELECT s.inst_id,
       p.name pdb_name,
       COUNT(*) sessions
FROM gv$session s
JOIN v$containers p
  ON s.con_id=p.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,p.name
ORDER BY p.name,s.inst_id;
Memory Usage per PDB and RAC Instance
SQL
SELECT
    s.inst_id,
    c.name pdb_name,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) pga_mb,
    COUNT(*) sessions
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;
Cluster-Wide Memory Distribution
SQL
SELECT
    c.name pdb_name,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) total_pga_mb,
    COUNT(*) sessions
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY c.name
ORDER BY total_pga_mb DESC;
RAC Workload Dashboard
SQL
SELECT
    s.inst_id,
    c.name pdb_name,
    COUNT(*) sessions,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) total_pga_mb,
    ROUND(SUM(p.pga_alloc_mem)/COUNT(*)/1024/1024,2) avg_pga_mb_per_session
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;

Application Continuity (AC)

Example configuration

Example: Server Control Utility (SRVCTL) manages Oracle RAC database services. In this guide, CDB01 is the Container Database (CDB), and CDB01_PDB01.paas.oracle.com is the Pluggable Database (PDB) service.

Bash
srvctl modify service \
-db CDB01 \
-service CDB01_PDB01.paas.oracle.com \
-failovertype TRANSACTION \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

Transparent Application Continuity (TAC)

Recommended configuration

Recommended configuration: SRVCTL applies the TAC service attributes to the CDB01 database and CDB01_PDB01.paas.oracle.com service used in this example.

Bash
srvctl modify service \
-db CDB01 \
-service CDB01_PDB01.paas.oracle.com \
-failovertype AUTO \
-session_state DYNAMIC \
-commit_outcome TRUE \
-replay_init_time 1800 \
-retention 86400

Best Practices

Operational checklist
  • Use SCAN exclusively.
  • Use services instead of instance-specific connections.
  • Enable FAN and Runtime Load Balancing.
  • Use Universal Connection Pool (UCP) with TAC.
  • Validate service distribution after scaling.
  • Test rolling maintenance procedures regularly.

Conclusion

A two-node Oracle RAC deployment on Exadata Exascale provides active-active scalability, workload balancing, and high availability. Combining SCAN, FAN, RLB, AC, and TAC delivers a resilient architecture for Oracle Database 26ai workloads.

Concepto de replay, componentes e implicaciones de memoria

Concepto de replayReplay significa que la pila cliente puede reconstruir una sesión recuperable y repetir una unidad lógica de trabajo segura después de una interrupción. SCAN y Fast Connection Failover (FCF) reconectan; Application Continuity (AC) y Transparent Application Continuity (TAC) agregan replay.
Componentes de replayAtributos de servicio configurados con SRVCTL, servicios Oracle RAC, listeners SCAN, Fast Application Notification (FAN), Oracle Notification Service (ONS), Runtime Load Balancing (RLB) y drivers o pools cliente compatibles con replay, como JDBC/UCP.
Requisitos de sesiónLas aplicaciones deben conectarse mediante servicios, no por instancias fijas. Después de falla de nodo o reubicación de servicio, las sesiones de reemplazo se establecen en instancias RAC sobrevivientes, por lo que los pools necesitan capacidad y reintentos suficientes.
Implicaciones de memoriaCada sesión reconstruida o reproducida consume memoria de proceso en la instancia destino. Durante failover, los nodos sobrevivientes pueden cargar más sesiones y memoria Program Global Area (PGA), User Global Area (UGA), cursores y estado de paquetes.
Validación operativaCompara gv$session, gv$process, v$containers y la ubicación de servicios después de escalar o ejecutar failover. Valida ubicación balanceada, sesiones aceptables por instancia y margen de memoria suficiente.

Mejores prácticas

Lista operativa
  • Use SCAN exclusivamente.
  • Use servicios en lugar de conexiones específicas por instancia.
  • Habilite FAN y Runtime Load Balancing.
  • Use Universal Connection Pool (UCP) con TAC.
  • Valide la distribución de servicios después del escalado.
  • Pruebe regularmente los procedimientos de mantenimiento rolling.

Conclusión

Un despliegue Oracle RAC de dos nodos en Exadata Exascale proporciona escalabilidad activo-activo, balanceo de carga de trabajo y alta disponibilidad. La combinación de SCAN, FAN, RLB, AC y TAC ofrece una arquitectura resiliente para cargas de trabajo de Oracle Database 26ai.

Conceito de replay, componentes e implicações de memória

Conceito de replayReplay significa que a pilha cliente pode reconstruir uma sessão recuperável e repetir uma unidade lógica de trabalho segura após uma interrupção. SCAN e Fast Connection Failover (FCF) reconectam; Application Continuity (AC) e Transparent Application Continuity (TAC) adicionam replay.
Componentes de replayAtributos de serviço configurados com SRVCTL, serviços Oracle RAC, listeners SCAN, Fast Application Notification (FAN), Oracle Notification Service (ONS), Runtime Load Balancing (RLB) e drivers ou pools cliente compatíveis com replay, como JDBC/UCP.
Requisitos de sessãoAs aplicações devem se conectar por serviços, não por instâncias fixas. Após falha de nó ou realocação de serviço, sessões substitutas são estabelecidas nas instâncias RAC sobreviventes, portanto os pools precisam de capacidade e novas tentativas suficientes.
Implicações de memóriaCada sessão reconstruída ou reproduzida consome memória de processo na instância destino. Durante failover, os nós sobreviventes podem carregar mais sessões e memória Program Global Area (PGA), User Global Area (UGA), cursores e estado de pacotes.
Validação operacionalCompare gv$session, gv$process, v$containers e o posicionamento de serviços após escala ou failover. Valide posicionamento balanceado, sessões aceitáveis por instância e margem de memória suficiente.

  • Use Universal Connection Pool (UCP) com TAC.
  • Valide a distribuição de serviços após o escalonamento.
  • Teste regularmente os procedimentos de manutenção rolling.

Conclusão

Uma implantação Oracle RAC de dois nós no Exadata Exascale fornece escalabilidade ativo-ativo, balanceamento de carga de trabalho e alta disponibilidade. A combinação de SCAN, FAN, RLB, AC e TAC entrega uma arquitetura resiliente para cargas de trabalho do Oracle Database 26ai.

Glossário

Estas definições resumem os acrônimos usados no artigo e apontam para documentação oficial da Oracle para referência mais detalhada.

AC Application Continuity

Alta disponibilidade com capacidade de replay para interrupções recuperáveis quando o replay está habilitado e é seguro.

CDB Container Database

Contêiner multitenant de banco de dados usado nos exemplos de serviço; os serviços PDB são executados dentro dele.

CRSCTL Controle de Clusterware

Utilitário de linha de comando usado para verificar e administrar recursos do Oracle Clusterware.

DNS Domain Name System

Camada de resolução de nomes usada por clientes para resolver nomes como SCAN.

ExaDB-XS Exadata Database Service on Exascale Infrastructure

Opção de serviço cloud da Oracle para Exadata Exascale Database Service no OCI.

FAN Fast Application Notification

Estrutura de notificação de eventos de alta disponibilidade do Oracle RAC para eventos planejados e não planejados.

FCF Fast Connection Failover

Comportamento de failover do cliente que ajuda a reconectar aplicações após mudanças de serviço ou instância.

OCI Oracle Cloud Infrastructure

Plataforma cloud da Oracle para executar recursos do Exadata Exascale Database Service.

ONS Oracle Notification Service

Transporte de notificações usado com eventos FAN para clientes conscientes de Oracle RAC.

PDB Pluggable Database

Contêiner de banco de dados dentro de uma CDB; este artigo usa CDB01_PDB01.paas.oracle.com como nome do serviço PDB.

RAC Real Application Clusters

Arquitetura Oracle em que várias instâncias em servidores de cluster acessam um banco de dados compartilhado.

RLB Runtime Load Balancing

Roteamento consciente de serviços que usa informações de carga de trabalho para direcionar conexões entre instâncias RAC.

SCAN Single Client Access Name

Nome estável de acesso ao cluster usado por clientes em vez de endereços específicos por nó.

SQL Structured Query Language

Linguagem de consulta usada aqui para inspecionar sessões RAC e distribuição de memória de processos.

SRVCTL Server Control Utility

Utilitário de linha de comando da Oracle para administrar bancos de dados RAC, serviços, listeners e recursos relacionados.

TAC Transparent Application Continuity

Modo de Application Continuity que rastreia automaticamente estado de sessão e limites de requisição.

TNS Transparent Network Substrate

Formato de descritor de conexão do Oracle Net usado para definir serviço, host, balanceamento de carga e failover.

UCP Universal Connection Pool

Pool de conexões JDBC da Oracle usado comumente com recursos de alta disponibilidade conscientes de RAC.

VIP Virtual IP

Endereço de rede administrado pelo cluster usado por listeners SCAN e caminhos de acesso de nó.
