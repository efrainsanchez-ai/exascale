Review TNS Aliases

Local naming uses tnsnames.ora aliases to map friendly service names to connect descriptors. Those descriptors define addresses, service names, failover options, and load-balancing behavior seen by clients.

Bash
Objective: Open the database-specific network admin directory and inspect the TNS aliases used by clients.
cd $ORACLE_HOME/network/admin/CDB01
cat tnsnames.ora

Section 10

Review Default Connection Service Configuration

The default PDB service is the baseline application entry point for this environment. Its SRVCTL attributes show current placement, failover type, connection load-balancing goal, and runtime load-balancing goal before recommended changes are applied.

Bash
Objective: Inspect the default PDB service attributes that become the baseline for failover and load-balancing changes.
srvctl config service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

This output shows that the default PDB service CDB01_PDB01.paas.oracle.com is enabled for the PRIMARY role and mapped to PDB01.

Cardinality: 2 with preferred instances CDB011,CDB012 means the service is placed on both RAC instances during normal operation; no available instances are listed for alternate placement.

Failover type: NONE, Failover method: NONE, TAF policy specification: NONE, Commit Outcome: FALSE, and Failover restore: NONE mean this default service is not configured for server-side TAF, Transaction Guard, or Application Continuity-style replay behavior.

Connection Load Balancing Goal: LONG favors session-count balancing for longer-lived connections, while Runtime Load Balancing Goal: NONE means the service is not publishing runtime load balancing advisory goals such as SERVICE_TIME or THROUGHPUT.

Default characteristics:

  • Failover type: NONE
  • Runtime Load Balancing Goal: NONE
  • Connection Load Balancing Goal: LONG

Section 11

Recommended OLTP Configuration (Memory Efficient)

OLTP services usually prioritize short transactions, predictable reconnect behavior, and low operational overhead. This section explains a session-failover configuration that keeps the existing service name and avoids creating extra database services.

Bash
Objective: Update the existing OLTP service with session failover retry behavior while preserving the same service name.
srvctl modify service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com \
-failovertype SESSION \
-failoverretry 30 \
-failoverdelay 5

Recommended values:

AttributeRecommended valueOracle documented values
Failover TypeSESSIONService-side SRVCTL -failovertype values: NONE, SESSION, SELECT, TRANSACTION, AUTO. Client-side TNS FAILOVER_MODE TYPE values are separate: session, select, none.
Failover Retries30Integer retry count.
Failover Delay5Integer delay in seconds.
Bash
Objective: Review the service configuration immediately after the modify command to confirm SRVCTL recorded the intended failover attributes before restarting the service.
srvctl config service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com
Bash
Objective: Stop, start, and check the existing service to clear prior runtime service configuration state and reload the updated failover attributes across the cluster.
srvctl stop service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

srvctl start service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

srvctl status service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

-failovertype SESSION changes the service from the captured default Failover type: NONE to session-level failover for supported clients. With NONE, the service does not request TAF-style session failover; a connection loss is returned to the application or pool to handle. With SESSION, the client can reconnect to an available instance and restore the database session context, but in-flight work is not replayed automatically.

-failovertype SELECT is the TAF option for read-only cursor continuity. For supported OCI clients, SELECT can let a session with open cursors continue fetching rows after reconnecting on another instance, while SESSION only recreates the session and does not attempt to recover select operations. SELECT adds normal-operation client-side overhead for select processing, so reserve it for workloads that benefit from continuing long fetches. With BASIC failover, the second node does not hold pre-established backup sessions; it must have the service available and enough CPU, memory, and process capacity to accept reconnects when failover occurs. Preconnected failover would be faster, but it allocates backup sessions in advance and requires the backup instance to support those extra connections.

Text
Objective: Provide the application-facing OLTP alias and equivalent JDBC Thin descriptor used by clients.

  • Place LOAD_BALANCE=ON and FAILOVER=ON directly under DESCRIPTION because they are connect descriptor options. LOAD_BALANCE randomizes the address path when multiple listener or SCAN addresses are available, and FAILOVER lets the client try another address if the chosen address is unavailable during connect time.

Section 13

Reporting Service Isolation

Reporting workloads are often read-heavy, longer running, and more likely to drive scans, sorts, temporary-space usage, and buffer-cache churn. A dedicated service gives that workload its own service name, placement policy, and monitoring identity without changing the PDB itself.

Bash
Objective: Create and validate a dedicated reporting service with preferred placement for long-running report workload isolation.
# Reporting-preferred RAC instance: CDB012
# Alternate instance for service failover: CDB011
srvctl add service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01_REPORTS.paas.oracle.com \
-pdb PDB01 \
-preferred CDB012 \
-available CDB011 \
-clbgoal LONG \
-rlbgoal NONE \
-failovertype NONE

srvctl start service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01_REPORTS.paas.oracle.com

srvctl status service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01_REPORTS.paas.oracle.com

srvctl config service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01_REPORTS.paas.oracle.com
Bash
Objective: Stop, start, and check the reporting service to clear prior runtime service configuration state and run with the final placement and service attributes.

-d $ORACLE_UNQNAME \

Section 15

Session and Memory Distribution Queries

Session and memory distribution is the runtime evidence for whether service placement is working as intended. GV$SESSION, GV$PROCESS, GV$SERVICES, and container metadata let the review compare connected workload and PGA use by instance, PDB, and service.

Run these queries from a DBA or monitoring account with access to GV$SESSION, GV$PROCESS, and V$CONTAINERS. Use the output after service changes, failover tests, or workload isolation changes to confirm that sessions and PGA memory are distributed as expected.

Sessions per Instance

SQL
Objective: Get a fast RAC-level count of sessions on each instance. Use this as the first balance check after service changes or failover testing to see whether one instance is carrying most of the connected workload.
SET LINESIZE 220
SET PAGESIZE 100
SET TRIMSPOOL ON
SET TAB OFF

COLUMN inst_id FORMAT 999
COLUMN sessions FORMAT 999999

SELECT inst_id, COUNT(*) sessions
FROM gv$session
GROUP BY inst_id;

Sessions per PDB and Instance

SQL
Objective: Break the session count down by PDB, RAC instance, and connection service name. This shows which service endpoint is placing sessions on each instance and helps confirm whether OLTP, reporting, or batch connections are landing where expected.
SET LINESIZE 220
SET PAGESIZE 100
SET TRIMSPOOL ON
SET TAB OFF

COLUMN inst_id FORMAT 999
COLUMN pdb_name FORMAT a30
COLUMN network_name FORMAT a45
COLUMN sessions FORMAT 999999

SELECT s.inst_id,
       p.name pdb_name,
       svc.network_name,
       COUNT(*) sessions
FROM gv$session s
JOIN v$containers p
  ON s.con_id=p.con_id
JOIN gv$services svc
  ON svc.inst_id=s.inst_id
 AND svc.con_id=s.con_id
 AND svc.name=s.service_name
WHERE s.type='USER'
GROUP BY s.inst_id,p.name,svc.network_name
ORDER BY p.name,svc.network_name,s.inst_id;

Memory Usage per PDB and RAC Instance

SQL
Objective: Estimate PGA memory allocated by user sessions for each PDB on each RAC instance. Use it to spot memory pressure, skewed service placement, or a workload class consuming more process memory on one node.
SET LINESIZE 220
SET PAGESIZE 100
SET TRIMSPOOL ON
SET TAB OFF

COLUMN inst_id FORMAT 999
COLUMN pdb_name FORMAT a30
COLUMN pga_mb FORMAT 999999990.99
COLUMN sessions FORMAT 999999

SELECT
    s.inst_id,
    c.name pdb_name,
    ROUND(SUM(p.pga_alloc_mem)/1024/1024,2) pga_mb,
    COUNT(*) sessions
FROM gv$session s
JOIN gv$process p
  ON s.inst_id=p.inst_id
 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;

Cluster-Wide Memory Distribution

SQL
Objective: Aggregate PGA usage by PDB across the whole RAC database. This highlights which PDB is the largest memory consumer regardless of instance placement and helps prioritize deeper session or SQL analysis.
SET LINESIZE 220
SET PAGESIZE 100
SET TRIMSPOOL ON
SET TAB OFF

COLUMN pdb_name FORMAT a30
COLUMN total_pga_mb FORMAT 999999990.99
COLUMN sessions FORMAT 999999

RAC Workload Dashboard

SQL
Objective: Combine session count, total PGA, and average PGA per session by PDB and instance. Use it as a compact workload dashboard to compare volume and memory intensity across RAC nodes before and after service configuration changes.
SET LINESIZE 220
SET PAGESIZE 100
SET TRIMSPOOL ON
SET TAB OFF

COLUMN inst_id FORMAT 999
COLUMN pdb_name FORMAT a30
COLUMN sessions FORMAT 999999
COLUMN total_pga_mb FORMAT 999999990.99

    s.inst_id,
    c.name pdb_name,

 AND s.paddr=p.addr
JOIN v$containers c
  ON s.con_id=c.con_id
WHERE s.type='USER'
GROUP BY s.inst_id,c.name
ORDER BY c.name,s.inst_id;

Section 16

Recommended OLTP Runtime Load Balancing Configuration

Runtime load balancing is a service advisory mechanism consumed by integrated connection pools. It helps new requests favor the instance currently offering the best service goal, but the application connection model must be compatible with pool-based routing.

Bash
Objective: Modify the existing OLTP service to publish connection and runtime load-balancing goals for integrated pools.
srvctl modify service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com \
-clbgoal SHORT \
-rlbgoal THROUGHPUT
Bash
Objective: Stop, start, and check the existing service to clear prior runtime service configuration state and reload the updated CLB and RLB attributes across the cluster.
srvctl stop service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

srvctl start service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

srvctl status service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

Validate the service configuration after the update. This confirms that SRVCTL reports Connection Load Balancing Goal: SHORT and Runtime Load Balancing Goal: THROUGHPUT for CDB01_PDB01.paas.oracle.com.

Bash
Objective: Confirm SRVCTL reports the expected connection and runtime load-balancing service attributes after the change.
srvctl config service \
-d $ORACLE_UNQNAME \
-s CDB01_PDB01.paas.oracle.com

You can also verify the applied service attributes from inside the database. Run this query from a DBA or monitoring account with access to GV$SERVICES; regular application users may not have privileges on this dynamic performance view. GV$SERVICES shows the service rows per RAC instance through INST_ID.

SQL
Objective: Validate the service load-balancing attributes from inside the RAC database by instance.
SET LINESIZE 220
SET PAGESIZE 100
SET TRIMSPOOL ON
SET TAB OFF

COLUMN name FORMAT a38
COLUMN connection_service_name FORMAT a45
COLUMN clb_goal FORMAT a10
COLUMN goal FORMAT a14

SELECT
  name,
  inst_id,
  network_name as connection_service_name,
  clb_goal,
  goal
FROM gv$services
WHERE name like '%.paas.oracle.com'
ORDER BY name, inst_id;
  • Connection Load Balancing Goal: SHORT
  • Runtime Load Balancing Goal: THROUGHPUT
AttributeRecommended valueDefault valueOracle documented values
Connection Load Balancing GoalSHORTLONGSHORT, LONG
Runtime Load Balancing GoalTHROUGHPUTNONENONE, SERVICE_TIME, THROUGHPUT

Use THROUGHPUT when the goal is to route work toward the instance completing work at the highest rate. Use SERVICE_TIME when response time for short OLTP transactions is the primary goal. Keep -clbgoal SHORT for integrated pools that use runtime load balancing advisories; LONG is mainly for closed or batch-style workloads.

No separate application-tier alias is required for this section. Runtime load balancing is controlled by service attributes, so existing aliases from section 9 or section 11 that resolve to CDB01_PDB01.paas.oracle.com inherit the updated service behavior.

Compared with the OLTP alias in section 11, the connection syntax can remain simpler here because runtime load balancing is driven by the service configuration and integrated pool advisories, not by adding a client-side FAILOVER_MODE block to the TNS descriptor.

Runtime load balancing works best for stateless request models where pooled connections can be borrowed, used for a short request, and returned to the pool. Enable Fast Connection Failover and use UCP, WebLogic, OCI, or ODP.NET pools that consume FAN and RAC load balancing advisory events.

For stateful applications, review before enabling THROUGHPUT globally. Long conversations, package globals, temporary tables, altered session state, open cursors, or affinity assumptions can reduce the benefit because the pool cannot freely route each new request. Move conversational state to the application tier or a table, reset session state when returning connections to the pool, mark request boundaries, and validate Application Continuity or Transparent Application Continuity coverage before relying on replay behavior.

Section 17

Simulate Node1 PDB Failure and Force Sessions to Node2

Use this controlled test when the application cannot drain or recycle connections cleanly. A non-forced service stop prevents new connections on CDB011 but can leave existing sessions visible there. To simulate a harder node1 failure, force-stop PDB01 on the node that hosts CDB011, then verify that the service remains available on CDB012 and that clients reconnect or replay work there.

Bash
Objective: Confirm where PDB01 and the default service are currently running before simulating node1 failure.
srvctl status pdb \
-db $ORACLE_UNQNAME \
-pdb PDB01 \
-detail

srvctl status service \
-db $ORACLE_UNQNAME \
-service CDB01_PDB01.paas.oracle.com \
-verbose

Confirm the current PDB and service placement before changing anything. This proves the test starts from the expected two-node service configuration.

SQL
Objective: Capture the session distribution baseline before forcing PDB01 to stop on node1.
SET LINESIZE 220
SET PAGESIZE 100
SET TRIMSPOOL ON
SET TAB OFF

-- Run from a DBA or monitoring account with access to GV$ views.
-- Execute once before the forced PDB stop to capture the baseline distribution.
-- Execute again after the forced PDB stop to compare the surviving sessions and active service placement.

COLUMN instance_name FORMAT a20
COLUMN service_name FORMAT a45
COLUMN active_service_name FORMAT a45
COLUMN status FORMAT a10
COLUMN sessions FORMAT 999999

SELECT s.inst_id, i.instance_name, s.service_name, s.status, count(*) as sessions
FROM gv$session s
join gv$instance i on i.inst_id = s.inst_id
WHERE s.type = 'USER'
  and s.service_name = 'CDB01_PDB01.paas.oracle.com'
GROUP BY s.inst_id, i.instance_name, s.service_name, s.status
ORDER BY s.inst_id, s.status;

SELECT a.inst_id, i.instance_name, a.name as active_service_name
FROM gv$active_services a
join gv$instance i on i.inst_id = a.inst_id
WHERE a.name = 'CDB01_PDB01.paas.oracle.com'
ORDER BY a.inst_id;

Run this before stopping PDB01. Keep the output as the reference distribution so the post-stop query can show exactly which sessions disappeared and where the active service remains.

-pdb PDB01 \
-node exasc011-ngkxu \
-stopoption IMMEDIATE \
-stopsvcoption IMMEDIATE \
-drain_timeout 0 \

Oracle documents srvctl stop pdb as stopping a PDB and its services. Use the -node option with the cluster node name, exasc011-ngkxu, not the database instance name CDB011. Use this disruptive command only for a controlled failure simulation when the application cannot drain connections. The -drain_timeout 0 value makes the operation immediate, and -force tells SRVCTL to force the PDB, its services, and dependent resources to stop on that node.

These errors are expected when active JDBC sessions are forcibly terminated. ORA-03113 means the client/server connection was broken, ORA-17008 means the JDBC connection object is already closed, and ORA-00028 means the database session was terminated. Why srvctl relocate service is not used here:

  • The observed PRCR-1145 / CRS-5702 result is expected for this configuration: the target node CDB012 already runs CDB01_PDB01.paas.oracle.com, so relocating the CDB011 service member to CDB012 would duplicate a member that is already active.

Best-practice notes:

  • Use the full SRVCTL keyword form in this section because Oracle documents single-letter SRVCTL parameters as deprecated and recommends full keywords for current scripts.
  • Existing sessions do not move as live sessions. Clients, pools, TAF, or Application Continuity must reconnect or replay according to their capabilities, and non-replayable in-flight work may fail and require application retry.
  • Run this in a controlled test window. A forced PDB stop is a failure simulation, not a graceful maintenance drain.
