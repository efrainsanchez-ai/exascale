RAC Health Check

Operational validation workflow for Grid Infrastructure, listener state, RAC database resources, services, SQL inventory views, and end-to-end connectivity.

Section 1

Grid Infrastructure Health Checks

Oracle Clusterware provides the cluster-aware resource framework used by RAC. CRSCTL is the Clusterware interface for checking cluster health, resource state, and node membership before database-level checks begin.

Bash
Objective: Validate Clusterware status, resource state, and node membership from the Grid Infrastructure owner before checking database resources.
sudo su - grid
crsctl check cluster -all
crsctl status resource -t
olsnodes
olsnodes -n
Operational note: olsnodes -i is deprecated.

Section 2

ASM Checks: Not Applicable to Exascale

In the Exadata Exascale path used by this guide, storage is presented through Exascale-managed infrastructure rather than customer-managed ASM instances. An absent ora.asm resource is therefore an expected characteristic, not automatically a fault.

Bash
Objective: Confirm the Exascale expectation that customer-managed ASM is not present for this deployment path.
srvctl status asm
Output
Objective: Expected output when the ASM resource does not exist in this Exascale path.
PRCR-1001 : Resource ora.asm does not exist

Section 3

Listener Health Checks

Oracle Net listeners receive client connection requests and route them to registered database services. In RAC, listener health is part of the path between SCAN or node addresses and the database service selected by the client.

Bash
Objective: Verify that the node listener resource is online on the target cluster node.
srvctl status listener -n <node_name>

Section 4

Database Health Checks

SRVCTL is the management utility Oracle expects for Oracle-managed RAC entities such as databases, instances, listeners, and services. Database status checks confirm whether each RAC instance is running under Clusterware control.

Bash
Objective: Load the database owner environment and confirm RAC database instance status through SRVCTL.
sudo su - oracle
. ./CDB01.env
srvctl status database -d $ORACLE_UNQNAME

Section 5

Database Configuration

The database configuration is the Clusterware metadata that describes how the RAC database is managed, including instance names, Oracle homes, startup policy, and placement information used by SRVCTL.

Bash
Objective: Review the database configuration registered in Oracle Clusterware, including instance placement and management attributes.
srvctl config database -d $ORACLE_UNQNAME

Section 6

Service Validation

RAC services are logical workload entry points. They decouple application connection names from individual instances and carry placement, failover, and load-balancing attributes used by clients and pools.

Bash
Objective: List service configuration and runtime status for the database, then check the default PDB service explicitly.
srvctl config service -d $ORACLE_UNQNAME
srvctl status service -d $ORACLE_UNQNAME

srvctl config service -d $ORACLE_UNQNAME -s cdb01_pdb01.paas.oracle.com
srvctl status service -d $ORACLE_UNQNAME -s cdb01_pdb01.paas.oracle.com

Section 7

SQL Health Checks

Dynamic performance views expose current database and instance state. In RAC, GV$ views return instance-scoped rows so the same query can compare status, services, and sessions across nodes.

View Inventory purpose
gv$instanceInstance-scoped RAC status and identity.
v$databaseDatabase-level state.
gv$servicesCluster-wide service metadata.
gv$active_servicesServices currently active by instance.
gv$sessionInstance-scoped session inventory.
v$servicesLocal service metadata.
v$pdbsPluggable database inventory and state.

Section 8

Connectivity Validation

Connectivity validation covers both name resolution and session creation. A TNS alias can resolve correctly while the final database service connection still fails, so both layers need to be checked.

Bash
Objective: Test client naming resolution with tnsping and validate an actual SQL*Plus connection.
tnsping CDB01_PDB01
sqlplus system@CDB01_PDB01

Operations

Exascale Operational Notes

  • Run Grid Infrastructure commands as the grid user.
  • Run database commands as the oracle user.
  • Use $ORACLE_UNQNAME for database-scoped SRVCTL checks.
  • Replace <node_name> with values from olsnodes.
